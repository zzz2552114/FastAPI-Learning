# 吸引大众用户的趣味后端实战项目指南

在掌握了 FastAPI 核心架构后，我们将目光转向**“普通大众用户”**。要吸引普通用户，项目必须具备三个特点：**概念有趣（好玩/治愈/沙雕）、交互极简（不需要您写复杂的前端页面）、具有病毒传播潜力**。

为了支撑这些看似简单的功能，您的后端（FastAPI + PostgreSQL + Redis + WebSockets）需要扛下极具挑战的高并发和复杂逻辑处理。

以下是为您量身定制的 3 个方向，技术栈完全包含在您所学的范围内，并且给出了极其详尽的 API 设计和避坑指南。

---

## 项目一：【治愈与匿名】赛博情绪漂流瓶 (Mood Capsule)

**概念介绍**：普通大众（尤其是年轻用户）非常需要匿名的情绪树洞。用户扔出一个包含心事的瓶子，后端通过轻量级 AI 识别情绪，然后推送给同样情绪低落需要共鸣的人，或者情绪高昂可以开导他的人。
**前端工作量**：极低。只需一个输入框（扔）和一个弹窗（捞）。

### 1. 技术选型与深层原因
*   **FastAPI + BackgroundTasks**：**为什么选？** 当普通用户疯狂倾诉时，我们必须接入免费的大模型 API（如 Kimi/DeepSeek）来提取情绪。外部请求极慢（1-3秒），如果直接写在路由里，您的网站会瞬间卡死。必须使用 `BackgroundTasks` 将 AI 处理推入后台，立刻给前端返回 "扔出成功"。
*   **Redis (List 结构)**：**为什么选？** 捞瓶子的逻辑是典型的队列操作。每次有人捞走一个，瓶子就应该从池子里消失。关系型数据库很难做这种高并发的“随机抽取并删除”，而 Redis 的 `RPOP` 命令是 O(1) 级别的，极速且天然原子化。
*   **PostgreSQL**：作为冷数据仓库，永久保存所有漂流瓶和回复的内容。

### 2. 极其详细的 API 路由设计

#### [路由 1] 倾诉心事（扔瓶子）
*   **路由**：`POST /api/v1/bottles/throw`
*   **请求 Body (JSON)**:
    ```json
    {
      "content": "今天明明很努力了，老板还是劈头盖脸骂了我一顿，真的好想哭...",
      "is_anonymous": true
    }
    ```
*   **细节校验 (Pydantic)**：`content` 必须使用 `Field(min_length=10, max_length=1000)`。如果不符合，利用之前学的全局异常拦截器，返回柔和的提示：`{"error": "情绪太短啦，多说一点吧~"}`。
*   **后端处理流程**：
    1.  生成随机 `bottle_id` (UUID)。
    2.  存入 PostgreSQL，状态为 `pending_ai`。
    3.  挂载后台任务：`background_tasks.add_task(process_bottle_mood, bottle_id, content)`。
    4.  立刻向前端返回响应。
*   **返回状态码与 Body**: `202 Accepted`, `{"message": "你的心事已经漂向大海...", "bottle_id": "xxx"}`

#### [路由 2] 捞取同频心事
*   **路由**：`GET /api/v1/bottles/catch`
*   **请求 Query (可选)**：`?target_mood=sad` (比如用户选择了想捞一个不开心的瓶子)
*   **后端处理流程**：
    1.  从 Redis 对应的分类列表（如 `mood:sad:pool`）中执行 `RPOP` 弹出一个 `bottle_id`。
    2.  如果没有取到，返回默认的“AI 治愈兜底瓶”。
    3.  拿 `bottle_id` 去 Postgres 中取出内容，剔除敏感字段后返回。
*   **返回状态码与 Body**: `200 OK`, `{"bottle_id": "yyy", "content": "今天明明很努力了...", "mood": "sad", "created_at": "..."}`

#### [路由 3] 暖心回复
*   **路由**：`POST /api/v1/bottles/{bottle_id}/reply`
*   **请求 Body**: `{"reply_message": "不要哭，明天会更好的！"}`

### 3. 大众涌入时必然遇到的坑与解法
*   **大坑 1：AI 接口由于并发过高导致挂掉，导致扔出去的瓶子永远没有情绪标签，卡死在系统里。**
    *   **终极解法 (融合新知)**：单独的 `BackgroundTasks` 不够用，因为它不会失败重试。您需要引入 **Tenacity** 库，在 `process_bottle_mood` 函数上加上 `@retry(stop=stop_after_attempt(3))`，让它失败后自动重试 3 次。如果 3 次还失败，捕获异常，默认给它打上一个 `mood=unknown` 的标签扔进 Redis，保证数据流通转。
*   **大坑 2：A 和 B 同时捞最后 1 个瓶子，A 捞到了但突然断网了，这个瓶子的心事再也没人看到了。**
    *   **终极解法**：不要使用 `RPOP`！改用 Redis 的高级命令 **`RPOPLPUSH`**（或者最新的 `LMOVE`）。将瓶子从“待捞池”转移到“正在被 A 看”的临时池。如果 A 在 3 分钟内没有调【路由 3】回复，一个定时任务会自动把瓶子从临时池踢回“待捞池”。

## 项目二：【沙雕共创】全网实时匿名“故事接龙” (Live Story Chain)

**概念介绍**：一帮无聊的网友在一个共同的页面里写小说，每个人只能写一句。如果不控制，场面绝对极度混乱爆笑。
**前端工作量**：一张纸（显示已有故事）+ 一个输入框 + 一个“抢笔”按钮。

### 1. 技术选型与深层原因
*   **FastAPI WebSockets**：**为什么选？** 因为这是“全网实时看接龙”。如果用 HTTP 轮询，1000 个人每秒发 1 次 GET 请求，服务器马上冒烟。WebSocket 可以维持长连接，只有当有人写了新句子时，后端才**主动广播**给这 1000 人。
*   **Redis 分布式锁 (SETNX)**：**为什么选？** 接龙的核心乐趣在于“抢夺写作权”。绝对不能允许两个人同时写第 5 句。必须在 Redis 里实现一把“笔（锁）”，谁抢到了这把锁，谁才有资格调接口写下一句。

### 2. 极其详细的 API 路由设计

#### [路由 1] 实时观影区 (WebSocket)
*   **路由**：`WS /ws/v1/story/live`
*   **功能**：用户进入页面即连接。后端只需维护一个 `active_connections` 列表。一旦有新句子诞生，遍历列表发送 JSON：`{"type": "new_sentence", "author": "网友A", "text": "就在这时，一只狗飞上了天。"}`

#### [路由 2] 疯狂抢笔
*   **路由**：`POST /api/v1/story/grab-pen`
*   **请求 Body**: `{"user_nickname": "沙雕网友A"}`
*   **后端处理流程**：
    1.  执行 Redis 命令：`SET pen_lock "沙雕网友A" NX EX 15`。
    2.  `NX` 保证只有没人拿笔时才能成功，`EX 15` 表示这把笔 15 秒后自动消失（防止抢了不写占茅坑）。
*   **返回状态码**:
    *   抢成功：`200 OK`, `{"message": "你抢到了笔！你有 15 秒的时间写下一句！"}`
    *   抢失败：`409 Conflict`, `{"error": "笔正在被【网友B】紧紧握着..."}`

#### [路由 3] 提交下一句
*   **路由**：`POST /api/v1/story/append`
*   **请求 Body**: `{"sentence": "就在这时，一只狗飞上了天。"}`
*   **后端处理流程**：
    1.  检查 Redis 里 `pen_lock` 的值是不是当前请求者，如果不是，抛出 403。
    2.  如果是，将句子存入 PostgreSQL。
    3.  立刻通过 WebSocket 向全网广播。
    4.  释放 Redis 的 `pen_lock`。

### 3. 大众涌入时必然遇到的坑与解法
*   **大坑 1：WebSocket 连接数达到上限导致 FastAPI 卡死。**
    *   **终极解法**：在路由函数内部，不能写阻塞的死循环。向所有客户端广播消息时，必须使用 `asyncio.gather(*[client.send_json(...) for client in clients])` 来并发发送，绝不能串行发送，否则第 1000 个人收到消息时黄花菜都凉了。
*   **大坑 2：抢笔的人太多，导致 API 接口被刷爆。**
    *   **终极解法**：利用已学的依赖注入 (`Depends`)，结合 Redis 的限流器。写一个前置依赖：如果同一个 IP 一秒钟内调了 10 次抢笔接口，直接在依赖层抛出 429 Too Many Requests 拦截，连业务逻辑都不要进。

## 项目三：【裂变测算】赛博灵魂抽卡机 (Cyber Soul Tarot)

**概念介绍**：大众对“测测你的XX”、“今日抽卡”毫无抵抗力。用户输入三个自己的随机特征，系统通过大模型结合概率，生成一张带有“奇葩属性数值”的灵魂卡片（比如：战斗力 5，摸鱼力 9999），极具分享属性。
**前端工作量**：一个表单 + 一个好看的 CSS 卡片展示。

### 1. 技术选型与深层原因
*   **极度严苛的 Pydantic 校验**：由于接入了 AI 生成，前端用户的输入如果不严格限制，AI 会被玩坏（比如输入 Prompt 注入攻击）。必须用 FastAPI 原生能力掐死非法字符。
*   **Redis 限流防刷**：这种裂变小游戏一旦在群里传开，瞬间会有上万请求。如果没有针对 IP 的限流，您的免费 AI 额度 5 分钟内就会枯竭。
*   **长轮询机制 (Long Polling)**：等待 AI 画卡片或生成文案需要时间。

### 2. 极其详细的 API 路由设计

#### [路由 1] 提交特征，开始施法
*   **路由**：`POST /api/v1/tarot/draw`
*   **请求 Body (极度受限)**:
    ```json
    {
      "trait_1": "爱喝咖啡",
      "trait_2": "不想上班",
      "trait_3": "夜猫子"
    }
    ```
*   **Pydantic 校验核心**：使用 `@field_validator` 配合正则表达式，确保每个词不能超过 5 个字，绝对不能包含英文字符（防 Prompt 注入），如果不符合，抛出 422 `{"error": "心不诚！请输入三个字的纯中文词汇。"}`。
*   **后端处理流程**：
    1.  校验通过。生成一个 `task_id`。
    2.  使用 `BackgroundTasks` 挂载 AI 调用任务。任务逻辑：把这三个词塞给 AI（Prompt：“根据这三个特征，生成一个搞笑的灵魂动物，并给出三个奇葩属性的分数”），拿到 JSON 结果后存入 Redis `task_result:{task_id}`，设置过期时间 1 小时。
*   **返回状态码与 Body**: `202 Accepted`, `{"message": "星象正在运转...", "task_id": "xxx-yyy"}`

#### [路由 2] 查询施法结果 (轮询接口)
*   **路由**：`GET /api/v1/tarot/result/{task_id}`
*   **后端处理流程**：
    1.  去 Redis 读取 `task_result:{task_id}`。
    2.  如果没有结果，说明 AI 还在跑。
*   **返回状态码**:
    *   未完成：`202 Accepted`, `{"status": "processing", "message": "还在跟宇宙通讯..."}`
    *   已完成：`200 OK`, `{"status": "done", "animal": "熬夜的猫头鹰", "stats": {"摸鱼力": 100, "发量": 2}}`

### 3. 大众涌入时必然遇到的坑与解法
*   **大坑 1：用户为了抽到“SSR 卡”，开启连点器疯狂点“抽取”，把 AI 额度和后台任务队列刷爆。**
    *   **终极解法**：在【路由 1】上挂载 `Depends(rate_limit_draw)` 依赖。利用 Redis，针对每个客户端 IP，限制 `1分钟内最多抽取 3 次`。超过则狠狠地抛出 HTTP 429 异常，带上嘲讽信息 `{"error": "天机不可频繁泄露，请休息一会儿再测。"}`。
*   **大坑 2：大模型 AI 胡言乱语，返回的根本不是 JSON，导致后端解析报错。**
    *   **终极解法 (运用 Pydantic 高级技巧)**：这是 AI 项目最容易崩的地方。不要直接信任 AI 返回的字符串。必须写一个 Pydantic 的 `AITarotResponse` 模型。拿到 AI 字符串后，尝试 `AITarotResponse.model_validate_json(ai_text)`。如果解析失败（`ValidationError`），立刻启动重试机制，或者用代码返回一个写死的“非酋史莱姆”兜底卡片，保证用户的接口永远能拿到正确的 JSON！
